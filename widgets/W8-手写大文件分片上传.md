# W8 · 手写大文件分片上传

> 业务轮子专项 · 需求 → 设计要点 → 核心实现 → 使用 → 进阶考点

## 需求场景

上传大文件(视频、镜像)时,单次上传慢、易超时、断了要重来。解决方案:**分片上传 + 并发 + 断点续传 + 秒传**。这是前端偏系统设计的高频题,能串起 W6(并发控制)、切片、哈希等能力。

## 设计要点

1. **分片(切片)**:用 `Blob.slice` 把大文件切成固定大小的小块;
2. **秒传**:算文件 hash,问服务端是否已存在,存在则直接完成(无需上传);
3. **断点续传**:问服务端已上传了哪些分片,只传缺失的;
4. **并发上传**:多个分片并发上传(用 W6 的并发控制);
5. **合并**:所有分片传完,通知服务端按序合并;
6. **进度**:统计已上传分片计算进度。

## 核心实现

### 1. 文件切片

```js
const CHUNK_SIZE = 5 * 1024 * 1024;   // 5MB 一片

function createChunks(file) {
  const chunks = [];
  let cur = 0;
  while (cur < file.size) {
    chunks.push({
      index: chunks.length,
      blob: file.slice(cur, cur + CHUNK_SIZE),   // Blob.slice 切片
    });
    cur += CHUNK_SIZE;
  }
  return chunks;
}
```

### 2. 计算文件 hash(秒传/断点的凭证)

```js
// 用 Web Worker + SparkMD5 计算,避免阻塞主线程(见 B5)
// 大文件可抽样哈希(首尾+中间片段)加速
async function calcHash(chunks) {
  return new Promise((resolve) => {
    const worker = new Worker('hash-worker.js');
    worker.postMessage({ chunks: chunks.map(c => c.blob) });
    worker.onmessage = (e) => resolve(e.data.hash);
  });
}
```

### 3. 完整上传流程

```js
async function uploadFile(file) {
  const chunks = createChunks(file);
  const hash = await calcHash(chunks);            // 文件指纹

  // ① 秒传 + 断点:问服务端状态
  const { uploaded, exists } = await checkUploaded(hash);
  if (exists) return console.log('秒传成功');      // 文件已存在,直接完成

  // ② 过滤掉已上传的分片(断点续传)
  const uploadedSet = new Set(uploaded);
  const tasks = chunks
    .filter(chunk => !uploadedSet.has(chunk.index))
    .map(chunk => () => uploadChunk(hash, chunk));

  // ③ 并发上传(复用 W6 的并发控制),带进度
  let done = uploadedSet.size;
  await asyncPool(4, tasks, async (task) => {
    await task();
    done++;
    onProgress(Math.round((done / chunks.length) * 100));
  });

  // ④ 通知服务端合并
  await mergeChunks(hash, file.name, chunks.length);
}

function uploadChunk(hash, chunk) {
  const form = new FormData();
  form.append('hash', hash);
  form.append('index', chunk.index);
  form.append('chunk', chunk.blob);
  return fetch('/upload/chunk', { method: 'POST', body: form });
}
```

## 使用示例

```jsx
<input type="file" onChange={e => {
  const file = e.target.files[0];
  uploadFile(file);   // 自动分片、秒传、断点续传、并发、合并
}} />
```

## 面试考点

**Q:大文件上传怎么设计?**
A:① 用 Blob.slice 把文件切成固定大小分片;② 计算文件 hash 作为唯一标识;③ **秒传**——上传前问服务端 hash 是否已存在,存在则直接完成;④ **断点续传**——问服务端已传了哪些分片,只传缺失的;⑤ 分片**并发上传**(控制并发数);⑥ 全部传完通知服务端**按序合并**。

**Q:秒传怎么实现?**
A:上传前计算整个文件的内容 hash(如 MD5),发给服务端查询。若服务端已有相同 hash 的文件,说明别人传过,直接建立引用返回成功,无需真正上传——即"秒传"。

**Q:断点续传怎么做?**
A:每个分片带上文件 hash 和分片 index 上传,服务端记录已收到的分片。续传时前端先查询该 hash 已上传的分片列表,过滤掉已传的,只上传缺失分片。刷新/断网后也能接着传。

**Q:计算 hash 会卡住页面吗?怎么优化?**
A:大文件全量算 hash 很耗时会阻塞主线程。优化:① 放 Web Worker 后台算(见 B5);② 抽样哈希——只取首尾和中间若干片段算(牺牲极小的准确性换速度);③ 用 requestIdleCallback 分片计算。

**Q:分片大小怎么定?**
A:权衡:太小则分片数多、请求开销大;太大则单片超时风险高、断点粒度粗。通常 2-10MB,可根据网络情况动态调整。

## 一句话总结

**大文件上传:Blob.slice 分片 → 算文件 hash(Web Worker/抽样,防阻塞)→ 秒传(hash 已存在直接完成)→ 断点续传(只传服务端缺失的分片)→ 并发上传(控制并发数+进度)→ 通知服务端按序合并;分片 2-10MB 权衡请求数与超时风险。**
