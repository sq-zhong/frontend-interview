# H3 · 浏览器渲染原理与重排重绘

> 热度:🔥🔥🔥

## 一句话考点

渲染流程:**DOM + CSSOM → 渲染树 → 布局(Layout)→ 绘制(Paint)→ 合成(Composite)**;**重排(reflow)**改变几何位置代价大,**重绘(repaint)**只改外观代价小,能用合成(transform/opacity)则不触发前两者。

## 原理精讲

### 1. 关键渲染路径(Critical Rendering Path)

1. **解析 HTML → DOM 树**:标签解析成节点树;
2. **解析 CSS → CSSOM 树**:样式规则树;
3. **合并 → 渲染树(Render Tree)**:只含可见节点(`display:none` 不入树,`visibility:hidden` 入树);
4. **布局 Layout(Reflow)**:计算每个节点的位置和尺寸;
5. **绘制 Paint**:把节点绘制成像素(颜色、边框、阴影);
6. **合成 Composite**:分层(GPU),合成到屏幕。

### 2. 重排(Reflow)

当元素的**几何属性**(宽高、位置、字体大小)改变,浏览器需重新计算布局。代价大,可能波及父/子/兄弟节点甚至整个文档。

**触发重排的操作**:
- 改变尺寸/位置(width、height、margin、padding、top、left);
- 增删可见 DOM 节点;
- 改变字体、内容;
- 读取会强制刷新布局的属性:`offsetTop`、`offsetWidth`、`scrollTop`、`getComputedStyle()`、`getBoundingClientRect()`(强制同步布局)。

### 3. 重绘(Repaint)

元素**外观**改变但不影响布局(如 color、background、visibility),跳过布局直接绘制。代价小于重排。

**重排必然引起重绘,重绘不一定引起重排。**

### 4. 合成(Composite)

`transform`、`opacity` 的变化可以只在合成阶段处理(GPU 加速),**跳过重排和重绘**,性能最好。

## 代码示例

### 减少重排的优化技巧

```js
// ❌ 频繁读写交替,触发多次强制同步布局(布局抖动)
for (let i = 0; i < items.length; i++) {
  items[i].style.width = box.offsetWidth + 'px';  // 读后写、写后读循环
}

// ✅ 读写分离:先批量读,再批量写
const width = box.offsetWidth;
for (let i = 0; i < items.length; i++) {
  items[i].style.width = width + 'px';
}
```

```js
// ✅ 批量 DOM 操作用 DocumentFragment
const fragment = document.createDocumentFragment();
for (let i = 0; i < 100; i++) {
  const li = document.createElement('li');
  fragment.appendChild(li);
}
list.appendChild(fragment);   // 只触发一次重排

// ✅ 先脱离文档流再改,或用 display:none 改完再显示
el.style.display = 'none';
// ...大量修改...
el.style.display = '';
```

```css
/* ✅ 动画用 transform/opacity,走合成层不触发重排重绘 */
.move { transition: transform 0.3s; }
.move:hover { transform: translateX(100px); }   /* 优于改 left */
```

## 高频追问

**Q:重排和重绘的区别?**
A:重排是几何属性变化导致重新布局,代价大;重绘是外观变化(颜色等)重新绘制,不涉及布局。重排一定伴随重绘,反之不一定。

**Q:如何减少重排重绘?**
A:① 批量修改(DocumentFragment、先 display:none);② 读写分离避免布局抖动;③ 用 class 一次性改样式而非逐条改;④ 动画用 transform/opacity;⑤ 用 `will-change` 提升为合成层。

**Q:为什么 transform 动画比改 left 流畅?**
A:transform 走合成阶段,由 GPU 处理,跳过重排和重绘;改 left 每帧都触发重排重绘,主线程压力大易掉帧。

**Q:`display:none` 和 `visibility:hidden` 对渲染树的影响?**
A:display:none 不生成渲染树节点(不占位),切换会触发重排;visibility:hidden 仍在渲染树占位,切换只触发重绘。

**Q:什么是布局抖动(layout thrashing)?**
A:在循环中交替"写样式-读布局属性",每次读都强制浏览器同步重新布局,导致大量重排。解决:读写分离。

## 一句话背诵版

**渲染:DOM+CSSOM→渲染树→布局→绘制→合成;重排=几何变化重新布局(贵)、重绘=外观变化(较轻)、重排必重绘;优化:批量改(fragment/display:none)、读写分离防抖动、动画用 transform/opacity 走 GPU 合成。**
