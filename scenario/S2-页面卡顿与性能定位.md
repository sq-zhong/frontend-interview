# S2 · 页面卡顿与性能定位

> 场景排错专项 · 现象 → 排查路径 → 常见根因 → 解决 → 举一反三

## 现象

页面滚动/输入/切换时明显卡顿、掉帧,交互反应迟钝,或某个操作后界面"冻住"一两秒。

## 排查路径(用工具,别猜)

```
① Performance 面板录制一段卡顿操作
   - 看主线程有没有"长任务"(Long Task,>50ms 的黄色块)
   - 看是脚本(黄)、渲染(紫)、绘制(绿)哪部分耗时
② FPS 曲线:是否掉到 60 以下、有没有红条(掉帧)
③ 定位长任务对应的函数(火焰图往下钻)
④ 若是渲染问题:开 Rendering → Paint flashing 看重绘区域
   - 大面积闪烁 = 重绘范围过大
⑤ React:用 Profiler 看哪个组件渲染耗时/渲染次数异常
   Vue:用 Vue DevTools 的性能面板
```

## 常见根因(分类)

**1. JS 长任务阻塞主线程**
- 一次性处理大数据(大数组循环、大 JSON 解析);
- 同步的复杂计算没有分片或放 Worker(见 B1/B5)。

**2. 频繁/无节制的渲染**
- 高频事件(scroll/resize/mousemove)直接触发重逻辑,没防抖节流(见 J10);
- React 父组件频繁 setState 导致大量子组件重渲染(见 R7);
- Vue 里 computed 写了副作用 / 大量深层 watch。

**3. 大量 DOM / 重排重绘**
- 长列表全量渲染(上千 DOM),没用虚拟列表(见 B7);
- 循环中读写交替触发布局抖动(见 H3);
- 动画用 left/top 而非 transform,每帧重排。

**4. 内存增长引发的卡顿**
- 内存泄漏导致 GC 频繁 / 页面越用越卡(见 S3)。

## 解决方案

| 根因 | 解决 |
|------|------|
| 长任务 | 时间分片(requestIdleCallback)、Web Worker、分页/懒计算 |
| 高频事件 | 防抖/节流、`passive` 事件监听 |
| 无效重渲染 | React.memo/useMemo/useCallback、缩小 state 粒度、拆组件 |
| 长列表 | 虚拟列表(react-window / vue-virtual-scroller) |
| 布局抖动 | 读写分离、批量 DOM、DocumentFragment |
| 动画卡 | 改用 transform/opacity 走 GPU 合成层 |

## 举一反三(面试如何答)

**Q:页面卡顿怎么定位和优化?**
> 先用 Performance 面板录制,看主线程有没有长任务、卡在脚本还是渲染。**如果是脚本长任务,考虑分片或 Web Worker;如果是渲染,查是不是无效重渲染或大量 DOM。**再针对性优化——高频事件加节流、列表上虚拟滚动、动画换 transform、React 用 Profiler 找出频繁渲染的组件。核心原则是**先测量定位、再对症下药,不凭感觉优化**。

**关键补充**:
- 强调**数据驱动**:先用工具量化(Performance/Profiler/FPS),而不是一上来就 useMemo 乱包;
- 提到 **INP 指标**(交互到下次绘制 <200ms)作为衡量交互流畅度的标准;
- 提"**长任务切片**"和"**主线程与 GUI 互斥**"能体现原理理解(见 B1)。

## 一句话总结

**卡顿定位靠 Performance 面板:先分清是 JS 长任务还是渲染问题;长任务用分片/Worker,无效重渲染用 memo+拆组件+缩 state 粒度,长列表用虚拟滚动,动画用 transform 走 GPU;原则是先测量后优化、用 INP 衡量交互。**
