# S5 · 发布后缓存不更新

> 场景排错专项 · 现象 → 排查路径 → 常见根因 → 解决 → 举一反三

## 现象

发布了新版本,但用户看到的还是旧页面;或部分用户白屏/报错(新旧资源混用);"清一下缓存就好了"——这正是需要根治的信号。

## 排查路径

```
① 硬刷新(Cmd+Shift+R)后是否正常?
   - 是 → 缓存问题无疑
② Network 面板看资源:
   - HTML 是 from disk cache / from memory cache?→ HTML 被强缓存了(大坑)
   - JS/CSS 请求的还是旧文件名(hash)?→ HTML 没更新导致引用旧资源
③ 是否出现 ChunkLoadError?
   - HTML 更新了引用新 hash 的 chunk,但 CDN 上新 chunk 还没同步 / 用户本地缓存了旧 chunk
④ 检查响应头:Cache-Control / ETag(见 N4)
```

## 常见根因(分类)

**1. HTML 被强缓存(最经典)**
入口 HTML 设了长 `max-age`,用户浏览器直接用本地旧 HTML,里面引用的还是旧版 JS,新版永远加载不���。

**2. 文件名没有内容 hash**
JS/CSS 用固定名(如 `app.js`)+ 强缓存,内容变了但文件名没变,浏览器继续用缓存的旧文件。

**3. CDN 缓存未刷新**
文件传上去了,但 CDN 边缘节点还缓存着旧文件,或新旧文件同步有时间差。

**4. Service Worker 缓存**
SW 缓存了旧资源,没有正确的更新策略(见 B6),用户一直拿缓存。

**5. 发布顺序问题**
先更新了 HTML(引用新 chunk),但静态资源还没传完 → 用户请求新 chunk 404(ChunkLoadError)。

## 解决方案(标准最佳实践)

```
静态资源(JS/CSS/图片):
  文件名带 contenthash(app.a1b2c3.js)
  + Cache-Control: max-age=31536000, immutable   ← 长期强缓存
  原理:内容变→hash 变→文件名变→自动请求新文件,旧的缓存自然失效

入口 HTML:
  Cache-Control: no-cache(每次协商缓存,拿最新)
  原理:HTML 小、必须实时,才能引用到最新的资源文件名

发布顺序:
  先传静态资源,确认 CDN 同步完成,再更新 HTML(避免引用到还没上传的 chunk)

CDN:
  发布后主动刷新/预热 CDN 缓存

Service Worker:
  正确的更新策略(skipWaiting + 提示用户刷新),或缓存加版本号
```

## 兜底:ChunkLoadError 自动恢复

```js
// 监听 chunk 加载失败,自动刷新一次拿新资源
window.addEventListener('error', (e) => {
  if (/Loading chunk .* failed/.test(e.message)) {
    // 防止死循环:用 sessionStorage 记录已刷新过
    if (!sessionStorage.getItem('chunk-reloaded')) {
      sessionStorage.setItem('chunk-reloaded', '1');
      location.reload();
    }
  }
});
```

## 举一反三(面试如何答)

**Q:发布后用户看到的还是旧页面,怎么解决?**
> 根因几乎都是缓存策略问题。标准做法是**"静态资源用 contenthash 命名 + 长期强缓存,入口 HTML 用 no-cache 协商缓存"**:内容一变文件名就变,自动加载新文件;HTML 不缓存保证能引用到最新的资源名。同时**发布顺序要先传资源再更新 HTML**,避免引用到还没上传完的 chunk 导致 ChunkLoadError,再配一个 chunk 失败自动刷新的兜底。

**关键补充**:
- 这题本质是 **N4 HTTP 缓存**的实战应用,能把强缓存/协商缓存讲清楚是加分;
- "清缓存就好"说明是缓存问题,别只让用户清,要从**发布机制**根治;
- 提 **CDN 刷新**和 **Service Worker 更新策略**体现完整性。

## 一句话总结

**发布不更新的根因是缓存:HTML 被强缓存或资源文件名无 hash;根治方案是静态资源 contenthash+长期强缓存(内容变文件名变自动更新)、HTML no-cache、发布先传资源再更 HTML、CDN 刷新、SW 更新策略,并加 ChunkLoadError 自动刷新兜底。**
