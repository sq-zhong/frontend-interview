# S4 · 请求问题排查(慢 / 竞态 / 失败)

> 场景排错专项 · 现象 → 排查路径 → 常见根因 → 解决 → 举一反三

## 现象

接口很慢、页面 loading 转圈;快速切换时显示了**旧的**数据(竞态);请求偶发失败;或"接口在 Network 里成功了,页面却没数据"。

## 排查路径

```
① Network 面板看这个请求:
   - Status:200 但数据不对?4xx/5xx?
   - Timing:慢在哪一段?
     · Queueing/Stalled 长 → 并发连接数满 / 优先级低
     · Waiting(TTFB)长 → 后端处理慢
     · Content Download 长 → 响应体太大
   - 是否发出了多个重复请求?
② 数据没渲染:请求成功但 UI 空?
   - 数据结构和预期不符(后端改了字段)
   - 竞态:后发的请求先返回,被先发的旧响应覆盖
③ 偶发失败:超时?跨域(见 N5)?401 循环?网络抖动?
```

## 常见根因 + 解决

### 1. 请求慢

| 慢在 | 原因 | 前端可做 |
|------|------|---------|
| TTFB | 后端慢 | 推动后端优化;加缓存;骨架屏改善感知 |
| 队列等待 | HTTP/1.1 并发限制(6 个) | 合并请求、HTTP/2、减少请求数 |
| 下载 | 响应体大 | 分页、字段裁剪、Gzip、按需加载 |
| 瀑布流 | 请求串行依赖 | 无依赖的并行 `Promise.all`(见 J8) |

### 2. 竞态(Race Condition)—— 高频且隐蔽

快速切换 tab/搜索时,请求 A 先发但后返回,覆盖了后发先返回的 B 的结果,显示旧数据。

```js
// ✅ 方案1:标志位丢弃过期响应
useEffect(() => {
  let cancelled = false;
  fetchData(id).then(res => {
    if (!cancelled) setData(res);   // 已切换则丢弃
  });
  return () => { cancelled = true; };
}, [id]);

// ✅ 方案2:AbortController 真正取消请求
useEffect(() => {
  const controller = new AbortController();
  fetch(url, { signal: controller.signal }).then(/*...*/);
  return () => controller.abort();
}, [url]);

// ✅ 方案3:只认最后一次请求(记录请求序号/时间戳)
```

### 3. 请求失败

| 失败 | 解决 |
|------|------|
| 超时 | 设置超时 + 失败重试(指数退避) |
| 401 | token 过期 → 无感刷新 token,注意避免刷新接口也 401 导致死循环 |
| 跨域 | 配置 CORS / 代理(见 N5) |
| 网络抖动 | 幂等请求自动重试;非幂等谨慎重试 |
| 重复提交 | 按钮 loading 禁用、请求去重 |

## 举一反三(面试如何答)

**Q:快速切换页面时数据错乱是什么问题?怎么解决?**
> 这是典型的**请求竞态**:多个请求并发,返回顺序不确定,后发的请求可能先返回、被先发的旧响应覆盖,显示了过期数据。解决有三种:**用标志位在清理时丢弃过期响应、用 AbortController 取消上一个请求、或只采纳最后一次请求的结果**。React 里通常在 useEffect 的清理函数里做。

**关键补充**:
- 竞态是**面试高频陷阱**,能主动说出来很加分;
- 请求慢要会看 **Timing 分段**(TTFB vs 下载 vs 排队),不同段对应不同优化;
- 提"**成熟方案**":用 React Query / SWR 这类库,内置了取消、去重、缓存、重试,不用手写。

## 一句话总结

**请求排错看 Network 的 Status 和 Timing 分段(TTFB 慢=后端、排队=并发限制、下载慢=体积);竞态用标志位/AbortController/取最后一次解决(高频陷阱);失败靠超时重试、401 无感刷新(防死循环)、去重防重复提交;成熟方案用 React Query/SWR。**
