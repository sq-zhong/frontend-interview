# D1 · 用 STAR 讲好项目经历

> 项目表达专项 · 为什么重要 → 框架 → 示例 → 常见坑 → 话术

## 为什么重要

面试官问"讲讲你做过的项目",这是**决定层级的关键时刻**。八股满分但项目讲不清,会被判"只会背、没实战"。同一个项目,平铺直叙 vs 结构化表达,给人的印象天差地别。

## STAR 框架

| 字母 | 含义 | 要点 |
|------|------|------|
| **S**ituation | 背景 | 项目是什么、你的角色、面临的问题(简短) |
| **T**ask | 任务 | 你要解决的具体目标/挑战 |
| **A**ction | 行动 | **你**做了什么(重点!技术方案、为什么这么选) |
| **R**esult | 结果 | **量化**的成果(数字!) |

**核心原则**:
- 重点在 **A(你的行动)和 R(量化结果)**,S/T 一两句带过;
- 说"**我**做了什么",不是"我们团队做了什么"(面试考的是你个人贡献);
- 结果**必须量化**:"首屏从 4s 优化到 1.5s""包体积减少 40%""错误率下降 60%"。

## 示例对比

**❌ 平铺直叙(听完不知道你强在哪):**
> "我做了一个后台管理系统,用的 React,有很多页面,我负责开发一些功能,还做了性能优化。"

**✅ STAR 结构化:**
> **(S)** ���负责公司数据中台的前端,这是个有 200+ 页面的大型后台,**(T)** 上线后用户反馈首屏加载要 5 秒多,严重影响体验,我主导了性能优化专项。
> **(A)** 我先用 Lighthouse 和 Performance 面板定位到三个瓶颈:首屏 bundle 过大、大量图表同步渲染、接口瀑布流。针对性做了三件事:① 路由级代码分割 + 组件懒加载,把首屏 JS 从 2.3M 降到 800K;② 图表用虚拟滚动 + 可视区懒渲染;③ 把串行接口改成 Promise.all 并行,并加了骨架屏。
> **(R)** 最终首屏从 5.2s 降到 1.8s,Lighthouse 性能分从 42 提到 89,用户跳出率下降了 25%。这套优化方案后来沉淀成团队的性能规范。

## 常见坑

1. **只讲技术栈罗列**:"用了 React、Redux、TS..." → 面试官要听的是你**解决了什么问题、怎么解决**;
2. **说"我们"不说"我"**:分不清你的个人贡献;
3. **没有量化结果**:"性能变好了"→ 好多少?没数字等于没说服力;
4. **平铺所有功能**:挑 1-2 个**有技术含量、有难点**的深讲,而非流水账;
5. **背稿子**:准备但别僵硬,要能接得住追问(见 D2)。

## 话术模板

准备项目时,每个项目按这个填空,练到能自然说出:

```
背景:我在 XX 项目负责 XX,当时遇到 XX 问题。
任务:我需要 XX(目标 + 约束)。
行动:我先 XX 定位/分析,发现 XX;然后我用 XX 方案,
      之所以选它是因为 XX(体现权衡)。过程中遇到 XX 难点,
      我通过 XX 解决。
结果:最终 XX(量化),并且 XX(额外价值,如沉淀规范/被复用)。
```

## 一句话总结

**用 STAR 讲项目:背景任务一句带过,重点讲"我"的行动(技术方案+为什么这么选+难点)和量化结果(必须有数字);挑 1-2 个有难点的深讲,别罗列技术栈、别说"我们"、别流水账;每个项目按模板练到能自然接住追问。**
