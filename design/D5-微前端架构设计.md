# D5 · 微前端架构设计

> 系统设计专项 · 题目 → 需求澄清 → 架构 → 难点与取舍 → 追问

## 题目

设计一个微前端架构,让多个团队独立开发、部署各自的前端应用,再集成到一个主应用里。

## 需求澄清

- **为什么用微前端**:大型应用拆分、多团队独立开发部署、技术栈无关、老系统渐进式迁移;
- **核心诉求**:① 独立开发/部署;② 应用间隔离(样式、JS);③ 主子应用通信;④ 技术栈无关。

## 主流方案对比

| 方案 | 原理 | 特点 |
|------|------|------|
| **iframe** | 天然隔离 | 隔离最彻底,但通信麻烦、体验割裂(弹窗/路由/白屏) |
| **single-spa** | 路由分发 + 生命周期 | 微前端鼻祖,需改造子���用 |
| **qiankun** | 基于 single-spa + sandbox | 阿里,开箱即用,主流 |
| **Module Federation** | Webpack5 运行时共享模块 | 去中心化,共享依赖好 |
| **Web Components** | 原生自定义元素 | 标准化,天然隔离 |

## 核心模块设计

### 1. 应用加载与路由分发

主应用(基座)根据路由,动态加载对应子应用的资源并挂载到指定容器;路由切换时卸载旧的、加载新的。子应用需暴露 `bootstrap/mount/unmount` 生命周期。

### 2. 隔离(核心难点)

**JS 隔离(沙箱)**:
- 快照沙箱:记录/恢复 window 状态(单实例);
- **Proxy 沙箱**:给每个子应用一个 Proxy 代理的 fake window,读写隔离(qiankun 用,支持多实例)。

**CSS 隔离**:
- Shadow DOM(彻底但有兼容/第三方弹窗问题);
- 动态样式表:子应用卸载时移除其 style;
- 约定 scoped 前缀 / CSS Modules。

### 3. 应用间通信

- 基座提供全局状态(如 qiankun 的 `initGlobalState`);
- 发布订阅 / 自定义事件;
- URL / 路由参数;
- 共享 localStorage(注意隔离)。

### 4. 公共依赖共享

避免每个子应用都打包一份 React/Vue:
- externals + CDN;
- **Module Federation** 运行时共享。

## 难点与取舍

**Q:qiankun 怎么实现 JS 隔离?**
A:用 Proxy 代理 window。子应用里对 window 的读写实际操作的是一个隔离的 fakeWindow,不污染真实 window;卸载时清理。多个子应用各有独立沙箱,互不干扰。

**Q:样式冲突怎么解决?**
A:① Shadow DOM 彻底隔离(但第三方组件挂到 body 的弹窗会逃逸);② 动态加载/卸载子应用的 style 标签;③ 加应用级 scoped 前缀。各有取舍,qiankun 默认用严格样式隔离(实验性)或前缀方案。

**Q:微前端的代价/缺点?**
A:① 复杂度上升(基座、沙箱、通信);② 首次加载子应用有开销;③ 公共依赖处理不好会重复打包;④ 调试和排错更难。**不是所有项目都该用**——小项目上微前端是过度设计。

## 追问

**Q:什么时候该用微前端?**
A:巨型应用多团队协作、需要独立部署、技术栈异构、老系统渐进迁移时。若是单团队中小项目,直接用 monorepo + 模块化更简单,别为了微前端而微前端。

**Q:iframe 方案为什么不常用?**
A:隔离虽好,但通信复杂(postMessage)、体验割裂(路由不同步、弹窗只能在 iframe 内、URL 无法共享、每次加载白屏),不适合体验要求高的集成。

**Q:Module Federation 和 qiankun 区别?**
A:qiankun 是"应用级"集成(加载整个子应用,重隔离);Module Federation 是"模块级"共享(运行时按需加载远程模块,重依赖共享、去中心化)。可结合使用。

## 一句话总结

**微前端解决多团队独立开发部署+技术栈无关:基座按路由分发加载子应用(bootstrap/mount/unmount 生命周期);核心难点是隔离(JS 用 Proxy 沙箱、CSS 用 Shadow DOM/动态样式/前缀)和通信(全局状态/发布订阅);公共依赖用 externals/Module Federation 共享;代价是复杂度上升,小项目别过度设计;主流 qiankun。**
