# D3 · 前端系统设计答题框架

> 系统设计专项 · 方法论:遇到"设计一个 XX"怎么答

## 为什么重要

"设计一个前端监控系统""设计一个低代码平台"——这类开放题没有标准答案,考的是**架构思维、权衡能力、全局观**。很多人一上来就写代码细节,方向就错了。系统设计要**先框架、后细节,先澄清、再设计**。

## 答题框架(五步法)

```
① 澄清需求(别急着答!)
   - 使用场景、规模(用户量/数据量)、核心指标、边界
   - 例:监控系统 → 要监控什么?错误?性能?行为?量级多大?

② 拆解核心模块
   - 把大问题拆成几个子系统,画出整体架构
   - 例:采集 → 上报 → 存储 → 分析 → 告警 → 展示

③ 逐模块设计关键点
   - 每个模块的技术方案、数据结构、流程

④ 讲权衡与难点(加分关键)
   - 每个决策的取舍:为什么这么选、代价、替代方案
   - 性能、可靠性、扩展性的考量

⑤ 演进与优化
   - MVP 先做什么、后续怎么扩展、如何应对规模增长
```

## 通用考量维度(每个系统都想想)

| 维度 | 要问自己 |
|------|---------|
| **性能** | 会不会影响主应用?采集/渲染开销? |
| **可靠性** | 出错了怎么办?数据会丢吗?降级方案? |
| **扩展性** | 加新功能容易吗?插件化? |
| **数据量** | 高频数据怎么处理?采样?批量?限流? |
| **兼容性** | 多端多浏览器?SDK 体积? |
| **安全** | 数据脱敏?权限?XSS/CSRF? |

## 示例开场(以"设计监控系统"为例)

> **(先澄清)** 在设计前我想先确认几点:我们主要监控错误、性能还是用户行为,还是都要?预期的接入应用规模和上报量级大概多少?对实时性要求高吗?
>
> **(再给架构)** 假设都要监控,我会把它拆成五层:**数据采集 SDK → 上报 → 服务端接收存储 → 分析聚合 → 告警 + 可视化**。前端重点在采集 SDK 和上报策略这两块……
>
> **(逐层展开 + 权衡)** 采集层要考虑不能影响主应用性能,所以……(见 D4)

## 常见坑

1. **不澄清就开答**:方向可能完全错,先问清楚需求和规模;
2. **陷入代码细节**:面试官要的是架构,不是让你现场写完整实现;
3. **只讲 happy path**:不提异常、降级、边界 → 显得不成熟;
4. **没有权衡**:每个方案直接给结论,不讲为什么、代价是什么 → 平庸;
5. **不结合规模**:小项目和亿级流量的方案完全不同,要按量级设计。

## 万能句式(体现架构思维)

- "在设计前我想先明确几个约束……"(澄清)
- "我会把它拆成 XX 几个模块……"(结构化)
- "这里有两种方案 A 和 B,考虑到 XX,我倾向 A,代价是 XX……"(权衡)
- "MVP 阶段先做 XX,规模上来后再 XX……"(演进)
- "这个模块的风险是 XX,我会用 XX 兜底……"(可靠性)

## 一句话总结

**前端系统设计五步:先澄清需求和规模(别急着答)→ 拆解核心模块画架构 → 逐模块设计关键点 → 讲权衡与难点(加分关键:为什么这么选、代价、替代方案)→ 演进优化(MVP+扩展);每个系统都从性能/可靠性/扩展性/数据量/兼容/安全六维度考量;先框架后细节,别陷入代码。**
