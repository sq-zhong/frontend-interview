# D2 · 项目深挖应对与技术亮点提炼

> 项目表达专项 · 为什么重要 → 框架 → 示例 → 常见坑 → 话术

## 为什么重要

STAR 讲完(见 D1),面试官会**顺着往下挖**:"为什么这么设计?""有什么代价?""还有别的方案吗?""遇到最大的困难是什么?"——这一层才真正区分候选人。深挖接不住,前面讲得再好也会露馅。

## 面试官深挖的四个方向

| 方向 | 典型问法 | 考察 |
|------|---------|------|
| **技术选型** | 为什么用 A 不用 B? | 权衡能力、是否真懂 |
| **难点** | 遇到最大的困难?怎么解决的? | 解决问题能力 |
| **代价/不足** | 这方案有什么缺点?如果重来? | 反思与深度 |
| **量化/原理** | 为什么快了这么多?底层原理? | 是否真做过、懂原理 |

## 亮点提炼:什么算"技术亮点"

不是"用了什么技术",而是**你的决策和创造**:

- **有权衡的选型**:"在 A 和 B 之间,考虑到 XX 约束我选了 A,代价是 XX,我用 XX 弥补";
- **解决了真问题**:线上事故排查、性能瓶颈、疑难 bug(结合 S 系列);
- **从 0 到 1 的设计**:搭了个工具/规范/轮子,被团队复用;
- **量化的改进**:任何带数字的提升。

## 示例:一次完整的深挖对话

> **面试官**:你说用了虚拟列表,为什么不用分页?
> **你**:产品要求是连续滚动浏览的体验,分页会打断;而数据量上万,全量渲染会卡。虚拟列表兼顾了连续体验和性能。**代价**是实现复杂、不定高时要处理高度测量,我用了预估高度 + ResizeObserver 测量缓存来解决。
>
> **面试官**:虚拟列表快在哪?原理说下。
> **你**:核心是 DOM 数量恒定——只渲染可视区的十几个节点,用占位元素撑总高度保证滚动条,translateY 定位。translateY 走 GPU 合成不触发重排,所以滚动流畅。(→ 能讲到 H3 合成层就是深度)
>
> **面试官**:如果让你重做,会改进什么?
> **你**:当时不定高的高度抖动处理得不够好,快速滚动偶有跳动。现在我会用更成熟的 react-virtual,或引入 IntersectionObserver 做更精确的可见性判断。

**这段为什么好**:每一层都接得住,有权衡("代价是...")、有原理(合成层)、有反思("会改进...")。

## 常见坑

1. **一问就卡**:说明项目不是自己主导的,或没深入思考过 → 只准备真正做过、想透的项目;
2. **只有优点没有缺点**:面试官问"有什么不足"时说"没什么缺点" → 显得没深度,任何方案都有代价;
3. **原理答不上**:"用了但不知道为什么快" → 平时就要往下挖一层;
4. **硬凹亮点**:把很普通的 CRUD 说成"很有挑战" → 反而减分,不如老实挑真正有含量的。

## 话术:应对不会的追问

不要不懂装懂。可以:
> "这块当时是 XX 同学主导的,我了解大致方案是 XX,但细节我没深入。不过如果让我设计,我会考虑 XX……" —— 诚实 + 展示思考,比乱编强。

## 一句话总结

**深挖考选型权衡、难点、代价、原理四个方向;亮点是"你的决策和创造"(有权衡的选型、解决真问题、从0到1、量化改进),不是技术罗列;每层要接得住:讲清为什么这么选、代价是什么、底层原理、如果重来怎么改;不懂就诚实说+展示思路,别硬凹别装懂。**
