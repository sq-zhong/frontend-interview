# R8 · 状态管理(Context / Redux / Zustand)

> 热度:🔥🔥

## 一句话考点

局部/低频共享用 **Context**;复杂全局状态用 **Redux**(单一 store、纯函数 reducer、单向数据流);轻量现代方案用 **Zustand**(hook 式、无样板代码)。

## 原理精讲

### 1. 何时需要状态管理库

- 多个不相关组件共享状态;
- 状态变化频繁、逻辑复杂;
- 需要时间旅行调试、中间件(异步、日志)。

简单场景优先 Context + useReducer,不必上重型库。

### 2. Redux 三大原则

1. **单一数据源**:整个应用一个 store;
2. **state 只读**:只能通过 dispatch **action** 改变;
3. **纯函数修改**:reducer 是 `(state, action) => newState` 的纯函数,不可变更新。

**数据流(单向)**:
```
View → dispatch(action) → reducer → 新 state → 更新 View
```

### 3. Redux 异步:中间件

reducer 是同步纯函数,异步逻辑放中间件:
- `redux-thunk`:action 可以是函数,内部做异步;
- `redux-saga`:用 generator 管理复杂副作用;
- **Redux Toolkit(RTK)**:官方推荐,内置 thunk、Immer(可写"可变"代码)、大幅减少样板。

### 4. Zustand 特点

- 用 hook 直接读状态,无需 Provider 包裹;
- 无 action/reducer 样板;
- 支持选择器订阅(只订阅用到的字段,精准更新);
- 体积小,上手快。

## 代码示例

### Redux Toolkit(现代写法)

```js
import { createSlice, configureStore } from '@reduxjs/toolkit';

const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: state => { state.value += 1; },  // Immer 允许"直接改"
    addBy: (state, action) => { state.value += action.payload; },
  },
});

export const { increment, addBy } = counterSlice.actions;
export const store = configureStore({ reducer: { counter: counterSlice.reducer } });
```

```jsx
// 组件中使用
import { useSelector, useDispatch } from 'react-redux';

function Counter() {
  const value = useSelector(s => s.counter.value);  // 订阅
  const dispatch = useDispatch();
  return <button onClick={() => dispatch(increment())}>{value}</button>;
}
```

### Zustand(轻量)

```js
import { create } from 'zustand';

const useStore = create(set => ({
  count: 0,
  inc: () => set(s => ({ count: s.count + 1 })),
}));

function Counter() {
  const count = useStore(s => s.count);   // 只订阅 count
  const inc = useStore(s => s.inc);
  return <button onClick={inc}>{count}</button>;
}
```

## 高频追问

**Q:Redux 的数据流是怎样的?**
A:单向。组件 dispatch 一个 action → reducer 根据 action 返回新 state(纯函数、不可变)→ 订阅的组件收到新 state 重渲染。

**Q:为什么 reducer 必须是纯函数?**
A:保证相同输入得到相同输出、无副作用,才能实现可预测的状态变更、时间旅行调试和易测试。异步/副作用交给中间件。

**Q:Redux 和 Context 区别?**
A:Context 是依赖注入机制,value 变化会让所有消费者重渲染,适合低频共享;Redux 是完整状态管理方案,有细粒度订阅(useSelector)、中间件、devtools,适合复杂高频状态。

**Q:Redux Toolkit 解决了什么?**
A:传统 Redux 样板代码多(action types、creators、switch reducer)。RTK 用 createSlice 自动生成 action、内置 Immer 支持"可变"写法、内置 thunk,大幅简化。

**Q:Zustand 相比 Redux 的优势?**
A:无需 Provider、无 action/reducer 样板、API 极简、内置选择器精准订阅、体积小。适合中小型项目或想减少模板代码的场景。

## 一句话背诵版

**低频共享用 Context;复杂全局用 Redux(单一 store、state 只读、reducer 纯函数、单向数据流、异步靠中间件、首选 RTK);轻量选 Zustand(hook 式、无 Provider、无样板、选择器精准订阅)。**
