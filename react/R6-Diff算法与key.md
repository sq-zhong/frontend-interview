# R6 · Diff 算法与 key

> 热度:🔥🔥🔥

## 一句话考点

React diff 基于**三个策略**把 O(n³) 降到 O(n):同层比较、类型不同直接替换、列表用 **key** 标识复用;key 应稳定唯一,**不要用 index** 作为会变动列表的 key。

## 原理精讲

### 1. 三大 diff 策略

1. **Tree diff(同层比较)**:只比较同一层级的节点,不跨层移动。若节点跨层级移动,React 会删除重建而非移动。
2. **Component diff(类型比较)**:
   - 类型相同 → 复用节点,只更新变化的属性;
   - 类型不同 → 直接销毁旧树、重建新树(即使内容相似)。
3. **Element diff(列表比较)**:同层的一组子节点通过 **key** 来标识,判断是新增、删除还是移动,从而复用。

### 2. key 的作用

key 帮助 React 识别哪些元素**变了、加了、删了、移动了**,精准复用而非重建。没有 key 时 React 按索引顺序对比,列表变动时容易错误复用。

### 3. 为什么不能用 index 作 key

当列表会**插入、删除、排序**时,index 会跟着位置变,导致:
- key 与内容错位,React 误判为"内容变了"而非"位置变了";
- 状态错乱(如输入框内容跟错行)、性能下降、动画异常。

**只有列表静态、不增删不排序时,用 index 才安全。**

## 代码示例

### 类型不同 → 整棵重建

```jsx
// 从 <div> 变成 <span>,即使子节点相同,React 也会销毁 div 子树重建
{isDiv ? <div><Child /></div> : <span><Child /></span>}
// Child 会被卸载重挂载,内部 state 丢失
```

### index 作 key 的经典 bug

```jsx
// ❌ 列表头部插入新项时出错
{list.map((item, index) => (
  <input key={index} defaultValue={item.name} />
))}
// 在开头插入一项后,所有 index 后移,React 复用了错误的 DOM,
// 导致输入框显示的内容与数据错位

// ✅ 用稳定唯一 id
{list.map(item => (
  <input key={item.id} defaultValue={item.name} />
))}
```

## 高频追问

**Q:React diff 的时间复杂度为什么是 O(n)?**
A:传统树 diff 是 O(n³)。React 用三个假设把它降到 O(n):① 只同层比较(不跨层);② 类型不同直接替换;③ 同层列表用 key 标识复用。

**Q:key 的作用是什么?**
A:作为同层子节点的唯一标识,帮 React 判断节点是复用、移动还是新增删除,减少不必要的 DOM 操作,并正确保持组件状态。

**Q:为什么不推荐用数组 index 作 key?**
A:列表增删或排序时 index 会变,key 与真实内容错位,导致 React 错误复用节点,引发状态错乱(如受控输入串行)、无效更新和动画异常。静态列表才可用 index。

**Q:两个不同类型的元素 diff 时会怎样?**
A:直接销毁旧节点及其整棵子树,重新创建新节点。所以要避免频繁切换根节点类型,否则子组件状态会丢失、性能变差。

**Q:key 相同但位置变化,React 怎么处理?**
A:React 认为是同一个元素发生了移动,复用其 DOM 和状态,只调整位置,这正是 key 的价值。

## 一句话背诵版

**Diff 三策略把 O(n³) 降到 O(n):同层比较、类型不同整树重建、列表用 key 复用;key 要稳定唯一,增删/排序的列表别用 index 作 key(会导致状态错位和错误复用)。**
