# N8 · TCP 三次握手与四次挥手

> 热度:🔥🔥🔥

## 一句话考点

TCP 建立连接需**三次握手**(确认双方收发能力),断开需**四次挥手**(双向独立关闭);TCP 面向连接、可靠传输,UDP 无连接、不可靠但快。

## 原理精讲

### 1. 三次握手(建立连接)

```
Client                          Server
  |------ SYN(seq=x) --------->|   ① 客户端请求建连
  |<-- SYN+ACK(seq=y,ack=x+1) -|   ② 服务器确认并请求建连
  |------ ACK(ack=y+1) ------->|   ③ 客户端确认
       连接建立(ESTABLISHED)
```

**为什么是三次(不是两次)**:
- 要**确认双方的发送和接收能力都正常**;
- 两次握手无法让服务器确认客户端能收到消息;
- 防止**失效的历史连接请求**突然到达服务器造成错误建连(浪费资源)。

### 2. 四次挥手(断开连接)

```
Client                          Server
  |------ FIN(seq=u) --------->|   ① 客户端请求关闭
  |<----- ACK(ack=u+1) --------|   ② 服务器确认(还能继续发数据)
  |<----- FIN(seq=w) ----------|   ③ 服务器数据发完,请求关闭
  |------ ACK(ack=w+1) ------->|   ④ 客户端确认
       等待 2MSL 后关闭
```

**为什么是四次**:TCP 是全双工,关闭要双向独立进行。服务器收到 FIN 后先回 ACK(此时可能还有数据要发),等数据发完再发自己的 FIN,所以中间的 ACK 和 FIN **不能合并**(而握手时可以合并成 SYN+ACK)。

### 3. 两个关键状态

- **TIME_WAIT**(主动关闭方):发完最后 ACK 后等 **2MSL**(报文最大生存时间的 2 倍)再关闭。原因:① 确保最后的 ACK 能到达对方(丢了可重发);② 让本次连接的残余报文在网络中消散,避免影响新连接。
- **CLOSE_WAIT**(被动关闭方):收到 FIN 回 ACK 后进入,若大量堆积说明程序没正确 close 连接。

### 4. TCP vs UDP

| | TCP | UDP |
|--|-----|-----|
| 连接 | 面向连接(握手) | 无连接 |
| 可靠性 | 可靠(确认重传、有序) | 不可靠(尽力而为) |
| 速度 | 较慢 | 快 |
| 场景 | HTTP、文件传输、邮件 | 视频/直播、DNS、游戏、QUIC |

## 高频追问

**Q:为什么握手是三次,不能两次?**
A:三次才能确认双方的收发能力都正常。两次的话服务器无法确认客户端能收到自己的响应;且能防止失效的历史 SYN 到达服务器导致错误建连、浪费资源。

**Q:为什么挥手是四次,不能三次?**
A:TCP 全双工,两个方向要独立关闭。服务器收到 FIN 后先回 ACK,但它可能还有数据要发送,等发完才发自己的 FIN。ACK 和 FIN 分两次发,无法像握手那样合并。

**Q:TIME_WAIT 是什么?为什么等 2MSL?**
A:主动关闭方发完最后 ACK 后进入 TIME_WAIT,等待 2MSL。目的:① 若最后 ACK 丢失,能收到对方重发的 FIN 并重发 ACK;② 保证旧连接的残留报文在网络中失效,不干扰使用相同端口的新连接。

**Q:大量 CLOSE_WAIT 说明什么?**
A:被动关闭方收到对方 FIN 后没有及时调用 close()。通常是程序 bug(连接没关闭、资源泄漏),会耗尽连接数。

**Q:TCP 如何保证可靠传输?**
A:序列号 + 确认应答(ACK)、超时重传、数据校验和、流量控制(滑动窗口)、拥塞控制(慢启动、拥塞避免)、数据排序去重。

## 一句话背诵版

**三次握手确认双方收发能力(防失效历史连接):SYN → SYN+ACK → ACK;四次挥手因全双工需双向独立关闭(ACK 与 FIN 不能合并):FIN→ACK→FIN→ACK,主动方 TIME_WAIT 等 2MSL 保证 ACK 送达+残余报文消散;TCP 可靠面向连接、UDP 快但不可靠。**
