# N2 · HTTP 版本演进(1.0 / 1.1 / 2 / 3)

> 热度:🔥🔥

## 一句话考点

HTTP/1.1 引入持久连接但有**队头阻塞**;HTTP/2 用**二进制分帧 + 多路复用 + 头部压缩**解决;HTTP/3 换用 **QUIC(基于 UDP)**彻底解决 TCP 层队头阻塞。

## 原理精讲

### HTTP/1.0
- 每个请求都新建 TCP 连接,用完关闭(开销大);
- 需手动加 `Connection: keep-alive` 才复用。

### HTTP/1.1(主流多年)
- **默认持久连接**(keep-alive),一个连接可发多个请求;
- 支持**管道化**(pipelining,但因队头阻塞实际很少启用);
- 新增缓存控制(Cache-Control)、range 断点续传、Host 头(虚拟主机)。
- **缺陷:队头阻塞**——同一连接中,前一个响应没返回,后面的请求要排队等待。浏览器只能靠开多个 TCP 连接(通常 6 个)缓解。

### HTTP/2
- **二进制分帧**:报文拆成二进制帧传输(此前是文本);
- **多路复用**:一个 TCP 连接上并行收发多个请求/响应(用 stream id 区分),解决了 **HTTP 层**队头阻塞;
- **头部压缩**(HPACK):压缩冗余的请求头;
- **服务器推送**(Server Push,已被废弃/少用);
- 请求优先级。
- **遗留问题**:仍基于 TCP,一旦丢包,TCP 要求按序,导致 **TCP 层队头阻塞**(所有 stream 都被阻塞)。

### HTTP/3
- 基于 **QUIC**(运行在 **UDP** 上),自己实现可靠传输和拥塞控制;
- 每个 stream 独立,**一个流丢包不影响其他流**,解决 TCP 层队头阻塞;
- **0-RTT / 1-RTT** 快速握手(集成 TLS 1.3,连接建立更快);
- 连接迁移(切换网络 IP 变化时连接不断)。

## 对比表

| 特性 | 1.1 | 2 | 3 |
|------|-----|---|---|
| 传输格式 | 文本 | 二进制帧 | 二进制帧 |
| 多路复用 | ❌(靠多连接) | ✅ | ✅ |
| 队头阻塞 | HTTP 层有 | TCP 层仍有 | ✅ 解决 |
| 头部压缩 | ❌ | HPACK | QPACK |
| 底层协议 | TCP | TCP | QUIC(UDP) |
| 握手 | TCP+TLS 多次 RTT | TCP+TLS | 0/1-RTT |

## 高频追问

**Q:HTTP/1.1 的队头阻塞是什么?**
A:同一 TCP 连接上请求需按顺序响应,前面的响应被阻塞会拖住后面所有请求。浏览器只能对同域名开多个(约 6 个)连接来缓解。

**Q:HTTP/2 如何解决队头阻塞?还有什么遗留问题?**
A:通过二进制分帧 + 多路复用,一个连接并行传多个流,解决了 HTTP 应用层的队头阻塞。但底层还是 TCP,丢包时 TCP 的按序重传会阻塞该连接上所有流,产生 TCP 层队头阻塞。

**Q:HTTP/3 为什么用 UDP?**
A:TCP 的队头阻塞是协议内核决定的,难以改。QUIC 基于 UDP 在应用层自己实现可靠传输,让各个流独立,单流丢包不影响其他流,并集成 TLS 1.3 实现更快握手和连接迁移。

**Q:HTTP/2 多路复用和 HTTP/1.1 多连接区别?**
A:1.1 靠开 6 个 TCP 连接并发,连接数有限、开销大;2 在单个连接上用 stream 并行无数请求,减少连接开销、更好利用带宽。

**Q:什么是头部压缩?**
A:HTTP 请求头有大量重复内容(cookie、UA 等)。HTTP/2 的 HPACK 用静态表+动态表+哈夫曼编码压缩头部,减少传输量。

## 一句话背诵版

**1.0 短连接;1.1 持久连接但有 HTTP 层队头阻塞(靠多连接缓解);2 二进制分帧+多路复用+HPACK 头部压缩(解决 HTTP 层阻塞,但 TCP 层仍有);3 基于 QUIC(UDP)让流独立、0-RTT 握手、连接迁移,彻底解决队头阻塞。**
