# N3 · HTTPS 与 TLS 握手

> 热度:🔥🔥🔥

## 一句话考点

HTTPS = HTTP + **TLS/SSL** 加密层;通过**非对称加密协商出对称密钥**,再用对称加密传输数据;数字证书解决公钥可信问题,防中间人攻击。

## 原理精讲

### 1. 为什么需要 HTTPS

HTTP 明文传输,存在:窃听(被抓包看到内容)、篡改(内容被改)、冒充(假服务器)。HTTPS 提供:
- **加密**:内容密文传输;
- **完整性**:防篡改(MAC/摘要);
- **身份认证**:证书验证服务器真实性。

### 2. 对称 vs 非对称加密

- **对称加密**(AES):加解密同一把密钥,快,但密钥怎么安全传给对方是难题;
- **非对称加密**(RSA/ECC):公钥加密、私钥解密,安全但慢。

**HTTPS 结合两者**:用非对称加密**安全地协商出一把对称密钥**,后续通信用对称加密(兼顾安全与性能)。

### 3. TLS 握手流程(TLS 1.2 简化)

1. **Client Hello**:客户端发送支持的 TLS 版本、加密套件、随机数 random1;
2. **Server Hello**:服务器选定加密套件,返回随机数 random2 + **数字证书**(含公钥);
3. **验证证书**:客户端用 CA 公钥验证证书合法性(见下);
4. **密钥交换**:客户端生成 pre-master secret,用证书公钥加密后发给服务器;
5. 双方用 random1 + random2 + pre-master 计算出**相同的对称密钥**(会话密钥);
6. 握手完成,之后用对称密钥加密通信。

> **TLS 1.3** 优化为 **1-RTT**(甚至 0-RTT),握手更快,废弃不安全算法。

### 4. 数字证书与信任链

- 服务器把公钥交给 **CA(证书颁发机构)**,CA 用自己的私钥签名生成证书;
- 浏览器内置受信任的 CA 根证书;
- 验证时用 CA 公钥验签,确认证书未被篡改、域名匹配、未过期;
- 防止中间人用假公钥冒充(因为伪造的证书没有可信 CA 的合法签名)。

## 代码示例

```
握手时序(TLS 1.2):

Client                                Server
  |------ Client Hello (random1) ------->|
  |<----- Server Hello (random2) --------|
  |<----- Certificate (公钥) ------------|
  |  验证证书合法性                       |
  |-- pre-master(用公钥加密) ----------->|
  |  双方计算会话密钥                     |
  |<========= 加密通信(对称) ==========>|
```

## 高频追问

**Q:HTTPS 加密过程简述?**
A:握手阶段用非对称加密协商出对称密钥:服务器发证书(含公钥),客户端验证后生成 pre-master 用公钥加密回传,双方结合随机数算出相同的会话密钥;之后用对称加密传数据。

**Q:为什么不全程用非对称加密?**
A:非对称加密计算慢、开销大,不适合大量数据传输。所以只用它安全地交换对称密钥,数据传输用高效的对称加密。

**Q:数字证书的作用?如何防中间人?**
A:证书由可信 CA 签名,绑定域名和公钥。客户端用内置 CA 公钥验签,确认公钥确实属于该网站。中间人无法伪造被 CA 合法签名的证书,所以无法冒充。

**Q:HTTPS 用的什么端口?比 HTTP 慢多少?**
A:HTTPS 默认 443,HTTP 默认 80。HTTPS 因握手多几次 RTT 和加解密有一定开销,但 TLS 1.3、会话复用、HTTP/2 已大幅优化,实际影响很小。

**Q:中间人攻击(MITM)是什么?HTTPS 如何防御?**
A:攻击者拦截通信、冒充双方转发。HTTPS 靠证书认证服务器身份 + 加密内容防御;只要用户不信任非法证书(不点"继续访问"),中间人无法解密或冒充。

## 一句话背诵版

**HTTPS = HTTP + TLS;非对称加密(证书公钥)协商出对称会话密钥,再用对称加密传数据(兼顾安全与性能);证书由 CA 签名、浏览器用 CA 公钥验签确认服务器身份,防窃听/篡改/中间人;端口 443,TLS 1.3 握手 1-RTT。**
