# AI6 · AI 辅助编码工程实践

> AI 时代专题 · 现状 → 用法 → 工程规范 → 边界 → 考点

## 背景

2026 年,AI 辅助编码(Copilot、Cursor、Claude Code 等)已是主流工作方式。面试越来越会问:**你怎么用 AI 提效?怎么保证质量?边界在哪?** 这考察的是**工程判断力**——会用是基础,知道何时不该信、如何校验才是水平。

## AI 编码的正确用法

| 场景 | 适合度 | 说明 |
|------|--------|------|
| 样板代码/CRUD | ✅ 很适合 | 重复性高、模式明确 |
| 单元测试生成 | ✅ 很适合 | 覆盖边界用例 |
| 正则/SQL/配置 | ✅ 适合 | 易错且有明确答案 |
| 代码解释/重构建议 | ✅ 适合 | 理解陌生代码 |
| 文档/注释 | ✅ 适合 | |
| 复杂业务逻辑 | ⚠️ 谨慎 | 需人把控正确性 |
| 架构决策 | ⚠️ 辅助 | AI 给选项,人做决定 |
| 安全敏感代码 | ❌ 必须人审 | 鉴权、加密、支付 |

## 工程规范(体现成熟度)

### 1. AI 生成的代码必须审查

**AI 是加速器不是负责人**。生成的代码要像 review 同事的 PR 一样审:
- 逻辑正确性、边界情况;
- 是否符合项目规范和现有模式;
- 有没有引入过时 API、安全漏洞、幻觉出的不存在的库;
- 性能和可维护性。

### 2. 上下文管理

给 AI 足够上下文(相关代码、约定、类型),它才能生成贴合项目的代码;用项目级规则文件(如 rules/AGENTS.md)固化团队规范。

### 3. 小步验证

让 AI 小步生成 + 及时测试,而非一次性生成大段难以验证的代码。配合类型检查、lint、测试形成防线。

### 4. 提示要具体

"写个函数"不如"写一个 TS 函数,输入 X 类型,处理 Y 边界,返回 Z,用项目现有的 request 工具"——上下文和约束越清晰,产出越可用(见 AI7)。

## 边界与风险(面试高光点)

**1. 幻觉**:AI 会编造不存在的 API、库、方法名,要验证;
**2. 过时**:训练数据有截止,可能给旧写法(如过时的框架 API);
**3. 安全**:可能生成有漏洞的代码(SQL 注入、XSS、硬编码密钥),敏感代码必须人审;
**4. 版权/合规**:生成代码的来源和授权;
**5. 能力退化风险**:过度依赖不理解原理 → 要保持"能看懂、能改、能判断对错"。

## 考点

**Q:你怎么用 AI 辅助编码?怎么保证质量?**
A:我把 AI 当加速器用在样板代码、测试、正则、重构建议这类重复或有明确答案的场景;复杂业务和架构决策让它给方案、我来判断。**核心是所有生成代码都要像 review 同事 PR 一样审查**——查逻辑、边界、是否符合项目规范、有没有幻觉出的 API 或安全问题,配合类型检查、lint、测试兜底,小步生成小步验证。

**Q:AI 生成代码有哪些坑?**
A:幻觉(编造不存在的 API/库)、过时写法(训练截止)、安全漏洞(注入、硬编码密钥)、不贴合项目规范。所以不能盲信,要验证、要审查,敏感代码必须人工把关。

**Q:AI 时代前端还需要懂原理吗?**
A:更需要。AI 能生成代码,但**判断对错、排查问题、做架构权衡、审查安全**都需要扎实的原理功底。不懂原理就无法识别 AI 的错误、无法在它卡住时接管。AI 抬高了"判断力"的价值。

**Q:如何让 AI 生成更贴合项目的代码?**
A:提供充分上下文(相关代码、类型、约定)、明确约束(用项目现有工具/风格)、用规则文件固化规范、小步迭代及时反馈。提示越具体产出越好。

## 一句话总结

**AI 辅助编码是加速器不是负责人:适合样板/测试/正则/重构建议,复杂业务和架构由人把控,安全代码必须人审;所有生成代码要像 review PR 一样审查(逻辑/边界/规范/幻觉/安全)+ 类型 lint 测试兜底 + 小步验证;风险是幻觉、过时、安全漏洞、能力退化;AI 时代更要懂原理——判断力和排错力才是核心价值。**
