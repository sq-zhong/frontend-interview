# B7 · 浏览器渲染优化:懒加载与虚拟列表

> 热度:🔥🔥

## 一句话考点

**懒加载**(按需加载可视区/未来资源)减少初始加载量;**虚拟列表**(只渲染可视区 DOM)解决长列表卡顿;两者核心都是"**只做当下必要的工作**"。

## 原理精讲

### 1. 懒加载(Lazy Loading)

延迟加载非首屏/暂不需要的资源,进入视口或需要时才加载。

**图片懒加载**:
- 原生 `loading="lazy"`;
- `IntersectionObserver` 监听是否进入视口(替代传统 scroll + getBoundingClientRect,性能更好,不占主线程)。

**组件/路由懒加载**:动态 `import()` 代码分割(见 B4)。

### 2. IntersectionObserver

异步观察目标元素与视口(或祖先)的交叉状态,进入/离开视口时触发回调。相比监听 scroll:
- 不阻塞主线程(浏览器底层实现);
- 无需手动计算位置和节流;
- 适合懒加载、无限滚动、曝光埋点。

### 3. 虚拟列表(Virtual List)

长列表(上千条)全部��染会产生大量 DOM,导致首次渲染慢、滚动卡、内存高。虚拟列表**只渲染可视区域内的少量 DOM**:
- 计算滚动位置,���出当前应显示的数据区间;
- 只渲染这部分 + 少量缓冲(buffer);
- 用占位容器撑起总高度,保证滚动条正常。

**关键计算**(定高):
```
起始索引 = Math.floor(scrollTop / itemHeight)
可见数量 = Math.ceil(viewportHeight / itemHeight)
偏移量 = 起始索引 * itemHeight  (用 transform/padding 撑位置)
```

不定高列表更复杂,需预估高度 + 动态测量缓存。

## 代码示例

### IntersectionObserver 图片懒加载

```js
const io = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {           // 进入视口
      const img = entry.target;
      img.src = img.dataset.src;
      io.unobserve(img);                  // 加载后停止观察
    }
  });
}, { rootMargin: '100px' });              // 提前 100px 预加载

document.querySelectorAll('img[data-src]').forEach(img => io.observe(img));
```

### 虚拟列表核心逻辑

```js
function VirtualList({ items, itemHeight = 50, viewportHeight = 500 }) {
  const [scrollTop, setScrollTop] = useState(0);

  const startIndex = Math.floor(scrollTop / itemHeight);
  const visibleCount = Math.ceil(viewportHeight / itemHeight);
  const endIndex = startIndex + visibleCount + 1;        // +buffer
  const visibleItems = items.slice(startIndex, endIndex);
  const offsetY = startIndex * itemHeight;

  return (
    <div style={{ height: viewportHeight, overflow: 'auto' }}
         onScroll={e => setScrollTop(e.target.scrollTop)}>
      {/* 撑起总高度,让滚动条正确 */}
      <div style={{ height: items.length * itemHeight, position: 'relative' }}>
        <div style={{ transform: `translateY(${offsetY}px)` }}>
          {visibleItems.map(item => (
            <div key={item.id} style={{ height: itemHeight }}>{item.name}</div>
          ))}
        </div>
      </div>
    </div>
  );
}
```

## 高频追问

**Q:图片懒加载怎么实现?**
A:① 原生 `loading="lazy"`;② IntersectionObserver 监听图片进入视口时把 data-src 赋给 src;③ 传统方案监听 scroll + getBoundingClientRect 判断位置(需节流,性能较差)。首选 IntersectionObserver。

**Q:IntersectionObserver 比监听 scroll 好在哪?**
A:它由浏览器底层异步实现,不占用主线程、无需手动计算元素位置和节流,天然高效。适合懒加载、无限滚动、元素曝光统计。

**Q:虚拟列表的原理?**
A:只渲染可视区域内的少量 DOM。根据滚动位置算出当前应显示的数据区间,只渲染这部分(加缓冲),用一个占位元素撑起列表总高度保证滚动条正常,滚动时动态更新渲染的数据和偏移。

**Q:虚拟列表定高和不定高的区别?**
A:定高可直接用 `scrollTop / itemHeight` 算索引,简单;不定高需预估高度或渲染后测量真实高度并缓存、动态修正偏移,实现复杂(如 react-virtualized 的 CellMeasurer)。

**Q:除了虚拟列表,长列表还有什么优化?**
A:分页/无限滚动(分批加载)、时间分片渲染、减少每项 DOM 复杂度、事件委托、图片懒加载、避免每项重复计算(memo)。

## 一句话背诵版

**懒加载按需加载(图片用 IntersectionObserver 进视口才加载、组件用动态 import),IntersectionObserver 不占主线程优于 scroll 监听;虚拟列表只渲染可视区 DOM(scrollTop/itemHeight 算区间 + 占位撑高度 + transform 偏移),解决长列表卡顿。**
