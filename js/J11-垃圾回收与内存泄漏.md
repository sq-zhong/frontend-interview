# J11 · 垃圾回收与内存泄漏

> 热度:🔥

## 一句话考点

JS 靠引擎**自动垃圾回收**,主流算法是**标记-清除**(从根对象出发,不可达的对象被回收);内存泄漏是指"不再使用却无法被回收"的内存,常见于意外全局变量、遗忘的定时器、闭包、脱离的 DOM 引用。

## 原理精讲

### 1. 内存生命周期

分配 → 使用 → 释放。前端主要关注"释放"由 GC 自动完成。

### 2. 两种经典 GC 算法

**引用计数(已淘汰)**:记录对象被引用次数,为 0 就回收。缺陷:**无法处理循环引用**(互相引用导致计数永不为 0)。

**标记-清除(现代主流)**:
1. 从根(全局对象、当前执行栈)出发,标记所有可达对象;
2. 清除未被标记的(不可达)对象。
- 天然解决循环引用:只要从根无法到达,即使互相引用也会被回收。

### 3. V8 的分代回收

- **新生代**(存活短的对象):Scavenge 算法,空间小、回收快。
- **老生代**(存活久的对象):标记-清除 + 标记-整理(解决内存碎片)。

## 代码示例

### 常见内存泄漏场景

```js
// 1. 意外的全局变量(未声明)
function leak() {
  name = 'oops';        // 忘记 var/let,挂到 window,不会被回收
}
// 严格模式可避免:'use strict'

// 2. 遗忘的定时器
const timer = setInterval(() => {
  doSomething(bigData);  // bigData 一直被引用
}, 1000);
// 组件卸载时必须 clearInterval(timer)

// 3. 脱离 DOM 的引用
const btn = document.getElementById('btn');
document.body.removeChild(btn);   // DOM 移除了
// 但 btn 变量仍引用它 → 无法回收,应 btn = null

// 4. 闭包长期持有大对象
function createHandler() {
  const largeData = new Array(1000000);
  return () => console.log(largeData.length); // largeData 常驻
}

// 5. 未解绑的事件监听
element.addEventListener('click', handler);
// 销毁前应 removeEventListener('click', handler)
```

### WeakMap / WeakSet 避免泄漏

```js
// WeakMap 的键是弱引用,键对象无其他引用时会被 GC 回收
const cache = new WeakMap();
function process(obj) {
  if (cache.has(obj)) return cache.get(obj);
  const result = compute(obj);
  cache.set(obj, result);   // obj 被回收时,这条缓存自动消失
  return result;
}
```

## 高频追问

**Q:引用计数为什么被淘汰?**
A:无法处理循环引用——两个对象互相引用,引用数永不为 0,即使外部已不可达也无法回收,导致泄漏。

**Q:如何排查内存泄漏?**
A:Chrome DevTools 的 Memory 面板:① Heap Snapshot 对比多次快照找持续增长的对象;② Performance 录制看内存曲线是否只涨不降;③ Allocation instrumentation 定位分配点。

**Q:WeakMap 和 Map 区别?**
A:WeakMap 键只能是对象且为弱引用,不阻止 GC,不可遍历、无 size;Map 键任意、强引用、可遍历。WeakMap 适合做缓存/私有数据,避免泄漏。

**Q:闭包一定造成内存泄漏吗?**
A:不会。闭包保留变量是设计意图,只有"不再需要却长期持有大对象"才算泄漏,用完置 null 即可释放。

## 一句话背诵版

**GC 自动回收,主流是标记-清除(从根不可达即回收,解决循环引用),V8 分新生代/老生代;内存泄漏五大场景:意外全局变量、遗忘的定时器、脱离 DOM 引用、闭包持有大对象、未解绑事件;缓存用 WeakMap 避免泄漏。**
