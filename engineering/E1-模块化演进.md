# E1 · 模块化演进

> 热度:🔥🔥

## 一句话考点

模块化解决全局污染、依赖管理问题;演进:**IIFE → CommonJS(Node,运行时同步)→ AMD(浏览器异步)→ ES Module(语言标准,静态、可 tree-shaking)**。

## 原理精讲

### 1. 为什么需要模块化

早期 JS 全局变量满天飞:命名冲突、依赖关系不清、难以维护。模块化提供**独立作用域**和**显式依赖**。

### 2. 演进历程

| 方案 | 时代 | 特点 |
|------|------|------|
| **IIFE** | 最早 | 立即执行函数创建私有作用域,手动挂全局 |
| **CommonJS** | Node.js | `require`/`module.exports`,**运行时同步加载**,值拷贝 |
| **AMD** | 浏览器 | `define`/`require`,**异步加载**(RequireJS) |
| **CMD** | 浏览器 | 依赖就近(SeaJS) |
| **UMD** | 通用 | 兼容 CommonJS + AMD + 全局 |
| **ES Module** | ES6 标准 | `import`/`export`,**编译时静态分析**,值引用 |

### 3. CommonJS vs ES Module(核心对比)

| 维度 | CommonJS | ES Module |
|------|----------|-----------|
| 加载时机 | 运行时(动态) | 编译时(静态) |
| 加载方式 | 同步 | 异步(浏览器) |
| 导出值 | **值的拷贝** | **值的引用**(动态绑定) |
| this | module.exports | undefined |
| Tree-shaking | ❌ 难 | ✅ 支持(静态结构) |
| 循环依赖 | 返回已执行部分 | 靠引用,处理更好 |
| 语法 | `require` / `module.exports` | `import` / `export` |

### 4. 为什么 ESM 能 tree-shaking

ESM 是**静态结构**——import/export 必须在顶层、模块名是字符串字面量,编译时就能确定依赖图和哪些导出未被使用,从而删除死代码。CommonJS 的 require 可以动态(在 if 里、拼接路径),无法静态分析。

## 代码示例

### CommonJS(值拷贝)

```js
// counter.js
let count = 0;
module.exports = { count, inc: () => count++ };

// main.js
const { count, inc } = require('./counter');
console.log(count);   // 0
inc();
console.log(count);   // 0(拷贝的值,不随原模块变化)
```

### ES Module(值引用)

```js
// counter.js
export let count = 0;
export const inc = () => count++;

// main.js
import { count, inc } from './counter.js';
console.log(count);   // 0
inc();
console.log(count);   // 1(引用绑定,反映最新值)
```

### 动态 import(ESM 也支持按需)

```js
// 返回 Promise,用于代码分割/懒加载
button.onclick = async () => {
  const { heavyFn } = await import('./heavy.js');
  heavyFn();
};
```

## 高频追问

**Q:CommonJS 和 ES Module 区别?**
A:① CJS 运行时同步加载、ESM 编译时静态分析;② CJS 导出值的拷贝、ESM 导出值的引用(动态绑定);③ ESM 支持 tree-shaking(静态结构),CJS 不行;④ 语法 require/module.exports vs import/export。

**Q:为什么 ESM 能做 tree-shaking 而 CommonJS 不能?**
A:ESM 的 import/export 是静态的(顶层、字面量),编译时可确定完整依赖图和未使用的导出,安全删除死代码;CommonJS 的 require 是运行时动态调用(可在条件里、路径可拼接),无法静态分析。

**Q:ESM 是同步还是异步?**
A:浏览器中 ESM 异步加载(`<script type="module">` 默认 defer)。模块解析、下载、执行都是异步的,不阻塞主线程。

**Q:CommonJS 的循环依赖会怎样?**
A:返回当前**已执行部分**的 exports(可能不完整)。因为 CJS 运行时执行,遇到循环时拿到的是模块尚未执行完的中间状态,可能得到 undefined。ESM 靠引用绑定,处理循环依赖更健壮。

**Q:UMD 是什么?**
A:Universal Module Definition,通用模块定义。通过判断环境,同时兼容 CommonJS、AMD 和浏览器全局变量,常用于需要在多环境运行的库(打包产物)。

## 一句话背诵版

**模块化演进:IIFE→CommonJS(Node,运行时同步、值拷贝、不能 tree-shaking)→AMD(浏览器异步)→ESM(ES6 标准,编译时静态、值引用、支持 tree-shaking);ESM 能 tree-shaking 是因为静态结构可分析依赖图;动态 import 用于按需加载。**
