# E7 · 构建性能优化

> 热度:🔥🔥

## 一句话考点

构建优化分两类:**打包速度**(缓存、多进程、缩小范围、用更快工具)和**产物体积**(压缩、tree-shaking、代码分割、按需引入、分析工具)。

## 原理精讲

### 1. 提升打包速度

| 手段 | 说明 |
|------|------|
| **持久化缓存** | webpack5 `cache: { type: 'filesystem' }`,二次构建复用 |
| **缩小 loader 范围** | `include/exclude` 限定 babel-loader 只处理 src,跳过 node_modules |
| **多进程** | `thread-loader`、esbuild-loader 并行编译 |
| **更快的工具** | 用 esbuild/swc 替代 babel-loader、terser |
| **resolve 优化** | 配置 extensions、alias、modules 减少解析开销 |
| **DllPlugin(旧)** | 预编译不变的第三方库(webpack5 后由缓存替代) |

### 2. 减小产物体积

| 手段 | 说明 |
|------|------|
| **压缩** | JS(Terser)、CSS(cssnano)、Gzip/Brotli |
| **Tree-shaking** | 删除未用代码(见 E4) |
| **代码分割** | 按需加载、抽公共 chunk(见 E4) |
| **按需引入** | lodash-es、组件库按需(babel-plugin-import) |
| **压缩图片** | image-webpack-loader、WebP |
| **externals/CDN** | 大库(vue/react)用 CDN 外链,不打进 bundle |
| **动态 polyfill** | 按目标环境注入,避免冗余 |

### 3. 分析工具

- `webpack-bundle-analyzer`:可视化各模块占用体积,定位大包;
- `speed-measure-webpack-plugin`:测量各 loader/plugin 耗时。

## 代码示例

### 缩小范围 + 缓存 + 多进程

```js
module.exports = {
  cache: { type: 'filesystem' },        // 持久化缓存,加速二次构建

  module: {
    rules: [{
      test: /\.js$/,
      include: path.resolve('src'),      // 只处理 src
      exclude: /node_modules/,           // 跳过依赖
      use: [
        'thread-loader',                 // 多进程
        'babel-loader',
      ],
    }],
  },

  resolve: {
    extensions: ['.js', '.ts', '.jsx'],  // 减少后缀尝试
    alias: { '@': path.resolve('src') }, // 加速路径解析
  },
};
```

### externals + CDN

```js
// 不打包 vue,改用 CDN
module.exports = {
  externals: {
    vue: 'Vue',                          // import vue 时用全局 Vue
  },
};
// HTML 里手动引入 CDN:<script src="https://cdn/vue.js"></script>
```

### 用 esbuild 压缩(更快)

```js
const { ESBuildMinifyPlugin } = require('esbuild-loader');
module.exports = {
  optimization: {
    minimizer: [new ESBuildMinifyPlugin({ target: 'es2015' })],  // 替代 Terser
  },
};
```

## 高频追问

**Q:如何加快 Webpack 构建速度?**
A:① 持久化缓存(webpack5 filesystem);② include/exclude 缩小 loader 处理范围;③ thread-loader 多进程;④ 用 esbuild/swc 替代 babel;⑤ 优化 resolve(extensions、alias);⑥ 合理拆分,减少重复构建。

**Q:如何减小打包体积?**
A:压缩(Terser/cssnano/Gzip)、tree-shaking、代码分割与懒加载、按需引入组件库、externals+CDN 外置大库、图片压缩、按需 polyfill;用 bundle-analyzer 定位大模块。

**Q:如何定位打包体积问题?**
A:用 webpack-bundle-analyzer 生成可视化图,查看哪些模块/依赖占用体积最大,针对性优化(按需引入、externals、拆分、替换大库)。

**Q:externals 有什么用?**
A:把指定依赖排除在打包外,运行时从外部(如 CDN 全局变量)获取。好处:减小 bundle 体积、利用 CDN 缓存和并行下载、多项目共享。代价是需保证外链可用。

**Q:为什么现在推荐用 esbuild/swc?**
A:它们用 Go/Rust 编写、多线程、无 JS 运行时开销,编译和压缩速度比 babel-loader、Terser 快数十倍。可作为 loader 或 minimizer 显著提升构建速度。

## 一句话背诵版

**构建优化两方向:提速(持久化缓存、include/exclude 缩范围、thread-loader 多进程、esbuild/swc 替代 babel、resolve 优化)和减体积(压缩、tree-shaking、代码分割、按需引入、externals+CDN、图片压缩);用 bundle-analyzer 定位大包、speed-measure 测耗时。**
